# FitNow 로드맵

## 현재 단계

FitNow는 쇼핑 홈, 주문/배송, 입점업체 관리, 총관리자 QA/정산 도구까지 한 번에 확인할 수 있는 프로토타입 단계입니다. 단순 화면 목업을 넘어 로컬 저장소와 Supabase 연동을 함께 검증하는 상태입니다.

## 배포 전 우선순위

1. 운영 QA 고정

- 관리자 QA 시나리오를 실제 운영 순서대로 한 번 더 정리합니다.
- 테스트 주문 생성, 배송 인증, 반품/환불, 정산, 리뷰 관리가 한 흐름으로 이어지는지 확인합니다.
- QA 결과를 저장하거나 복사할 수 있는 방식은 유지합니다.

2. 고객 홈 완성도

- 홈 상품 카드, 배지, 추천 영역, 룩/마이 이동 흐름을 계속 다듬습니다.
- 모바일 화면에서 버튼, 텍스트, 카드 간격이 겹치지 않는지 확인합니다.
- 고객이 상품을 보고 주문까지 가는 흐름을 우선합니다.

3. 입점업체 운영 화면

- 들어온 주문, 처리할 주문, 반품/환불 요청의 버튼 가독성을 계속 유지합니다.
- 업체가 어떤 주문을 먼저 처리해야 하는지 바로 보이게 합니다.
- 반품/환불 권한은 입점업체와 총관리자가 모두 처리할 수 있게 유지합니다.

4. 배송 인증과 리뷰 사진

- 배송 인증 사진은 고객도 확인할 수 있어야 하므로 공개 읽기 정책을 유지합니다.
- 배송 완료 후 사진 보관 기간은 현재 정책상 30일 기준입니다.
- 리뷰 사진은 등록, 교체, 삭제, 관리자 숨김 처리를 안정화합니다.

5. 배포 안정성

- `check:deploy`와 `check:e2e`를 배포 전 필수 확인으로 유지합니다.
- 로컬 확인은 `02_PREVIEW_BUILD_AND_OPEN.bat`를 기준으로 합니다.
- GitHub Actions 성공 후 배포 주소에서 한 번 더 확인합니다.

## 이후 고도화 후보

- 상품 등록/수정 관리자 화면 고도화
- 입점업체별 정산 내역 다운로드
- 고객 리뷰 신고/블라인드 사유 관리
- 주문 상태별 알림 문구 정리
- 실제 결제 PG 연동 준비
- Supabase RLS 정책을 운영 권한 기준으로 더 세분화
- 대용량 `src/standalone-app.js`를 기능별 모듈로 분리

## 냉정한 판단

앱 배포 자체는 가능한 수준에 가까워졌지만, 실제 운영용으로는 QA 시나리오, 권한 정책, 결제/정산 책임 범위가 더 명확해야 합니다. 다음 큰 작업은 화면 추가보다 운영 중 문제가 생겼을 때 확인하고 되돌릴 수 있는 관리 기능을 단단히 하는 쪽이 맞습니다.
